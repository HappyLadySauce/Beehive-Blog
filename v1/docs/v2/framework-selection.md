# Beehive Blog v2 微服务框架选型建议

## 1. 选型目标

v2 的框架选型，重点不是“哪家更流行”，而是：

- 是否适合 Go 团队长期维护
- 是否适合微服务拆分
- 是否方便做 API、RPC、任务、鉴权、配置、可观测性
- 是否方便承载搜索、索引、AI、MCP 这些非传统博客能力

## 2. 当前推荐结论

### 主推荐

如果你们明确要采用 Go 微服务，并且希望尽快起盘，**推荐优先考虑 `go-zero`**。

原因：

- 在中文团队里上手成本较低
- REST、RPC、配置、服务治理、限流等能力比较完整
- 适合中后台和业务 API 快速建设
- 对“网关 + 多服务 + worker”结构比较友好

### 备选方案

如果你们更重视：

- 更自由的分层
- 更偏工程化的控制力
- 更强的长期可塑性

可以考虑：

- `Kratos`
- `Go-kit`

不过从当前阶段来看，这两者都比 go-zero 更吃团队工程经验。

## 3. 各方案简评

## 3.1 go-zero

优点：

- 脚手架成熟
- RPC + REST 配合自然
- 内建较多微服务常用能力
- 适合快速搭出业务骨架
- 对中国开发者资料更友好

不足：

- 项目风格会更偏框架约束
- 某些高度定制场景没有那么轻盈

适合你们的原因：

- v2 要尽快落地多服务雏形
- 需要兼顾 API、任务、鉴权、检索、AI 接入
- 需要比纯手写框架更快形成工程规范

## 3.2 Kratos

优点：

- 分层和可扩展性更强
- 社区对工程化和整洁架构支持更自然
- 适合长期演进

不足：

- 相对更偏“会的人用起来很舒服”
- 起盘速度未必比 go-zero 更快

适合场景：

- 团队已经熟悉 DDD、clean architecture、分层设计
- 希望把服务边界和工程模式控制得更细

## 3.3 Go-kit

优点：

- 足够底层
- 适合完全定制

不足：

- 工程门槛高
- 对当前项目而言偏重

结论：

不建议作为 v2 当前首选。

## 3.4 不使用完整微服务框架，自己搭

优点：

- 灵活
- 不受框架约束

不足：

- 早期成本高
- 很多治理能力要自己补
- 容易导致每个服务风格不统一

结论：

除非团队非常强，不建议一开始就完全手搭。

## 4. 推荐技术组合

如果以 `go-zero` 为主，建议组合如下：

- 微服务框架：`go-zero`
- API 网关：`go-zero gateway` 或自建 gateway 服务
- 服务通信：`zrpc`
- 数据库：`PostgreSQL`
- 缓存：`Redis`
- 搜索：`Meilisearch` 或 `Elasticsearch`
- 任务队列：可先基于 Redis / DB 事件实现，后续升级消息队列
- 观测：Prometheus + Grafana + Loki/ELK

## 5. 推荐的工程原则

即使使用 go-zero，也建议坚持这些原则：

- 框架只负责基础设施，不主导领域模型
- 领域模型和业务规则不要散落在 handler 里
- 事件和异步任务独立出来
- 搜索、RAG、MCP 等能力保持清晰边界

## 6. v2 当前建议

当前我建议直接把文档口径定为：

1. v2 采用微服务架构
2. 框架主选 `go-zero`
3. 保留 `Kratos` 作为技术评审备选
4. 服务先按领域划分，避免过细拆分
5. 搜索服务、AI 服务、MCP 服务作为独立边界设计

## 7. 最后结论

如果目标是：

- 尽快建立 v2 微服务骨架
- 让团队可以稳定推进
- 避免框架层投入过重

那么目前最实际的选择是：

**v2 优先采用 `go-zero`。**

如果未来团队工程能力进一步增强，或者某些服务确实不适合 go-zero，再局部引入其他实现方式也可以，但现在不建议在框架选择上摇摆太久。
